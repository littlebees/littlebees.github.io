這次來規劃要做的圖書館管理系統要有什麼功能

用user story來寫寫接下來要做的圖書館管理系統

User Story

As a “role”, I want to “do something”, so that I can “get value”.

就是 主詞 透過 動詞 可以 受詞，用來定義使用者需要什麼來達成什麼

像，

  • 作為 一般人，可以註冊成讀者，來使用圖書館
  • 作為 館員，可以上架新書，來增加館藏
  • 作為 讀者，可以尋找書籍，來確認是不是自己需要的書
  • 作為 讀者，可以查看書籍的庫存訊息，來了解目前館藏狀態
  • 作為 讀者，可以預約書籍，來確保一定可以拿到書
  • 作為 讀者，可以預約書籍，來確保一定可以拿到書
  • 作為 讀者，可以借閱書籍，來把書帶回去看
  • 作為 館員，可以通知快逾期/逾期的讀者，來把書還回去
  • 作為 館員，可以對違規讀者處罰
  • 作為 館員，可以對通知讀者預約書籍到達，來把書借回去
  • 作為 館員，可以把書放回架上，來讓書可以被看到

因為這個結構很簡單，所以開發與非開發人員都可以理解，但是也因為太簡單，所以

  1. 動詞與受詞可能
  • 包的太大，讀者可以把書帶回去看
  • 包得太細(其實這就是task)，讀者可以在填完資料後，按下送出，得到預約的資格
  1. Story之間的先後順序不明顯
  2. Story之間的關係(結構)不明顯

所以我們除了story還要一個組織以及分割story的手段

INVEST

雖然說是評斷story的手段，但這些點也是分割story的方向

  • 獨立（Independent）：必須有採取行動的可能性，而且本身是「可完結」的，不能與其他Story有相依性。
  • 可修改（Negotiable）：只要還沒結束，Story就必須可以重寫，要預留修改的餘地。
  • 有價值（Valuable）：必須實際為顧客、使用者或利害關係人傳遞價值。
  • 可估算（Estimable）：必須能掌握大小長短。
  • 規模小（Small）：必須小到能夠預估且易於規劃。如果太龐大就必須重寫或拆解成多個Story。
  • 可測試（Testable）：必須有一個必須通過的考驗，通過之後才算完成。

User Story Mapping

組織story的手段

  1. 列出要做的事，前面的動作要是後一個動作的鋪成
  • 上架新書
  • 尋找書籍
  • 查看書籍的庫存訊息
  • 讀者註冊
  • 預約書籍
  • 借閱書籍
  • 快逾期/逾期通知
  • 歸還書籍
  • 對違規讀者處罰
  • 通知預約書籍到達
  • 把書放回架上
  1. 針對每一件要做的事，細化出完成它所需的步驟

  1. 進行優先順序的取捨

  1. 仔細思考每件事之間可能的階段關係
