SOAP & RESTful

把SOAP與RESTful用PL的角度其實很清楚

SOAP需要caller知道有什麼method可以用，其實就是【物件的方法調用】
RESTful每個東西都有自己的id，同時還有state只在調用api後才有可能改變，這就是【array】

like ___ in PL the amount of method(verb) state relation between caller & callee data type
RESTful Array 7 server master & slave JSON/XML…
SOAP obj method invoke depend on design server & client depend on design XML

RESTful & graphql

graphql就是SQL的變種，
可以預想之後會變得越來越像SQL，
就像C++的smart pointer與各個語言中的lambda function一樣

graphql的最大好處是【可以預先知道回傳回來的資料裡面有什麼】
如果是RESTful要等到server回傳parse後才能知道有什麼item，再根據資料再發request，十分的浪費頻寬
這是前端的勝利!!

但是，是後端的不幸
後端頓時要做與DB一樣的事，除了從原本的DB拉資料，再透過商業邏輯轉換後，還要組合出graphql給定的格式!!

後端複雜度上升，同時快取也不好處理，每次的request都是不同的結構，這樣要如何選定快取的key?
