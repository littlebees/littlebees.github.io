Rails的測試Survey

Posted on 2020-10-25 Edited on 2022-04-12 In Web , Rails , FAQ

動機

寫測試之前先看怎麼寫測試

怎樣的程式叫{一樣}?

程式碼一樣，就是一樣。
但我們不可能知道所有程式碼，所以可以

  1. 看是不是可以從A程式藉由公理改寫到B程式 (Program derivation)
  2. 只看兩個程式的輸入輸出是否一樣 (Behavioral equivalence)

我們可以把輸入輸出抽象(type)，再配合公理，看是不是可以得到想要的type (static analysis)

實際跑看能不能拿到想要的值(testing)

怎麼觀察程式? (怎麼設計test)

  1. 他會吃什麼? type，形式…
    a. 在 input type中的理想或是最好處理的input
    b. 在 input type中的edge case
    c. 在 input type中的不合法、不被處理的input
    d. 不是input type的input
  2. 他會給什麼? type，形式…
    a. 合法輸入的結果
    b. 非法輸入的結果: 錯誤訊息或是有規律的輸出
  3. 使用會發生什麼事 (會影響到其他人嗎)
  4. 程式的輸入、輸出、狀態是怎麼對應到目的?
  5. 使用前有什麼前提
    a. 這程式都是在特定情況下被呼叫? (spesific or general)
    b. 被call的環境、順序 (call stack、userstory)

前四點都是寫測試內容要考慮到的，第一點也決定要不要用假物件，最後一點是如何切分測試的依據

從內部看就是用call stack去規劃，如果是rails就有一個是

  1. model : 只測試model (in Rails Guide: Model Testing)
  2. model controller : 測試{一個}controller (in Rails Guide: Functional Tests)
  3. model controller+ : 測試{多個}controller (in Rails Guide: Integration Testing)
  4. model controller+ view : 測試view (in Rails Guide: System Testing)

從外部看就是依據使用情境來規劃

小總結一下

  • 從裡面: 內部狀態與流程如何對應到外部輸入與輸出
  • 從外面: 輸入與輸出是不是符合預期

從裡面看的測試就是 Unit test 與 integration test
從外面看的測試就是 system test 與 acceptance test

假物件就是只會給出特定值得物件，通常是取代其他需要會產生任意input或是花時間的input

術語

主要是Rspec in rails的術語，畢竟

  1. 這是rails相關的文章，同時
  2. Rspec有許多自己的術語，

要先把他與rails guide上與一般術語之間的關聯找出來

  • Rspec術語 => Guide上的術語 (一般聊天時用的術語)
  • Feature spec => System Testing (acceptance test OR end-to-end test)
  • Controller specs => Functional Tests (如果model測完就叫unit test，沒測完就叫integration test)
  • Request spec => Integration Testing (integration test)
  • Model specs => Model Testing (unit test)

還有testing中會用到假物件，因為像io或是db之類的處理可能會花許多時間，但我們只在意他的output，所以可以用假物件模擬output，這是他的主要精神(還有滿足interface要求的空值也算假物件)，但不知道為什麼奇怪的術語有點多，所以把相關文章放在參考

Capybara : interface to browser

By default Capybara will only locate visible elements.
This is because a real user would not be able to interact with non-visible elements.

基本動作(user的動作)

  1. 開網頁: visit(post_comments_path(post))
  2. 點擊: click_link('id-of-link'); click_button('Save'); click_on('Link Text')
    a. 處理跳出視窗:accept_alert { click_link('Show Alert') }; dismiss_confirm { click_link('Show Confirm') }
  3. 輸入
    a. 文字: fill_in('First Name', with: 'John')
    b. 檔案: attach_file('Image', '/path/to/image.jpg')
    c. checkbox: check('A Checkbox'); uncheck('A Checkbox')
    d. radiobox: choose('A Radio Button')
    e. droplist: select('Option', from: 'Select Box')
  4. 有我要的內容嗎: expect(page).to have_content('foo')

比較偏開發者

  1. DOM related
    a. 有DOM嗎: page.has_selector?('table tr'); expect(page).to have_css('table tr.foo')
    b. 拿到DOM: find_field('First Name').value; find_link(class: ['some_class', 'some_other_class'], :visible => :all).visible?
  2. 在一定區域內做事: within_fieldset('Employee') { fill_in 'Name', with: 'Jimmy' }
  3. (debug) screenshot & current HTML: save_and_open_page; print page.html; page.save_screenshot('screenshot.png')
