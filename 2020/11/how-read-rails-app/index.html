怎麼讀一個新到手的Rails App

Posted on 2020-11-05 Edited on 2021-02-20 In Rails重生計畫 Disqus:

動機

說好要寫測試，但是在寫測試之前要先懂程式在幹嘛

所以有了這篇

流程

  1. 實際跑跑看，至少把基本操作跑一次
  2. 看 db/schema.rb 看關聯與table
  3. 看 model(或test) 看邏輯 (照理來說，所有重要的邏輯都要在這裡，test可以用來印證猜想與確認使用方式)
  4. 看 routes.rb 確認可以用的action
  5. 看 controller(與view!?)

Schama

從repo上的schema.rb可以畫出下面的圖，其中

  • 雙箭頭虛線是 many-to-many
  • 實線箭頭是 belong_to，箭頭指到的是該table持有的id

其中詭異的是，

  1. receipt與bill都是 many-to-many，但為什麼會有兩個
  2. search為什麼要belong_to publisher??

Model

copy state & receipt&bill

單看receipt與bill都是與copy和user有關，所以應該是借單之類的角色
但實際分工就要看model了，觀察model得到以下狀態圖

想吐槽當時的自己，

  1. 其實receipt與bill可以和在一起，再多一個狀態就好
  2. 這很明顯是有限狀態機，應該有更好的解法
  3. 這能做的事會不會太少，如果書弄丟了的狀態是，但是之後找回來的話?

search’s publisher id

當初疑惑為什麼要有publisher id，現在答案揭曉了。
需要publisher id作條件去找，不過這裡能支援的search是不是有點少?

結論

重新設計一個圖書館管理系統，用

  • 有限狀態機
  • TDD
  • Rails6
  • Solr (search engine, instead of using simple SQL)
    實作後端

用

  • erb
  • vue.js
    實作前端，但先用erb撐一下

用agile的方式設計spec(我沒上過軟體工程，一直認為軟體工程打高空的成分太多，所以不怎麼喜歡，但現在這個side project可以試著導入)

所以到時候的開發流程可能是

  1. Spec
  2. Design Schama
  3. Design model(& finite state machine)
  4. Design controller/view
  5. Design frontend with vue.js
