動機

現在backend已經能動了，雖然有許多可以改進的地方…

但還是先讓他動起來，所以先來看前端框架

JSX

大概結構像

JS中有HTML，HTML中有JS，但要包起來

同時區分html tag與component的方式是第一個字有沒有大寫

class & hook

class 是在同一元件下的所有狀態共用同一個life circle(所以一旦很多事要做，會很擠)，

然而hook會自己附到相對應的life circle階段，如果state之間互像影響會讓code跳來跳去的，變得不好看

整體而言(資料的傳遞方式)

一路向下，資料都是從parent到child，都用attr傳到props中，child要改變上層的狀態只能透過

  1. 來自parent的callback來改變狀態
  2. context來傳遞

但這樣很verbose，也會很混亂，不好trace，都是因為資料只能一路向下，對此重構的方式有

  1. 從源頭解決: 把state往上抽、high-order-component、props’ render function
  2. 抽到全域: flux

可控與不可控元件

資料的路徑都是

  1. event會碰到來自parent的callback
  2. callback會setstate自己(parent)的state
  3. 最後透過props去render

但是，兩個差在資料的來源不同

可控

  1. event會碰到來自parent的callback (input從event來)
  2. callback會setstate自己(parent)的state
  3. 最後透過props去render

不可控

  1. event會碰到來自parent的callback
  2. callback會setstate自己(parent)的state (input從ref來)
  3. 最後透過props去render

因為ref被用來綁dom，所以如果是要從很上層傳ref下去會很痛苦，所以有forwardref，其實就是用function去接ref，而不是用props

大概的演化流程

View + Controller(ViewModel, representation model) => View + Controller + Model => View + Route + Controller + Model

一開始view與邏輯是和在一起，後面為了管理許多資料，先把view與邏輯分出來，變成view(presentation component)與controller(container component)，最後把狀態抽到同一個地方，成為model

Flux

就是有限狀態機，在MVC中的model

大概就是

  1. 把所有邏輯放到狀態機中(dispatcher, reducer)
  2. 把狀態與資料放在全域中，這樣大家都看的到，不使用callback或是ref(store)
  3. 用subscribe的方式來讓view知道store更新

這裡就比較貼近原本的MVC，整個流動是一個三角形，
而web的MVC(model2)，就是一個向上的尖尖，model與view之間的溝通都要透過controller完成

Redux

基本上就是Flux，但

  1. 只有一個store，所以要拿到狀態與action，要去切出適合的部份給component，沒辦法直接subscribe，所以有所謂的connect
  2. state是直接放在參數(如果再改一下，就可以用state monad去包整個reducer)，不像其他是直接改全域變數

component

資料從哪裡來?

  1. props
  2. state
  3. context

props: 從caller來的資料，就是一般函數的參數
state: component自己的資料，就是物件的attribute
context: 可以當成全域變數，但修改值時會限定在一定區域，見Racket的parameter
